Vibe Coding, the Pro Way
Welcome. Honest frame up front: we will NOT finish an app tonight — you'll leave with the PROCESS. Quick hands: who's vibe-coded before? Who's been frustrated by AI coding? (Builds rapport.) Tonight I drive Claude Code, you watch; we build a real little app — the conference vendor QR scanner — as the vehicle.
02
What is vibe coding?
Define it simply. Emphasize: vibe coding ≠ specs-to-code-and-ignore-the-code. The code is still the battleground. You're the architect; AI is the very fast junior who never sleeps. Tonight shows the disciplined version.
03
What is an LLM?
Keep it concrete for data folks: it's a probabilistic model — same family as the prediction stuff you already know, just enormous. Two properties drive everything tonight: (1) it reasons in the text you give it (context = its working memory), (2) it forgets between sessions. Those two facts shape the whole workflow.
04
Who is "this Claude fellow"? (+ get Claude Code)
Distinguish the model (Claude) from the agent (Claude Code). Claude Code is what makes it 'vibe coding' vs 'chatting' — it actually edits the repo and runs things. LIVE MOMENT: switch to the browser, go to claude.com/claude-code, show how simple it is to install (one command) and that it runs in your terminal / editor. This is my daily driver and what we'll drive all night.
05
Smart zone vs dumb zone
This is the spine. Credit: Dex Horthy / Human Layer, popularized by Matt Pocock. As context fills, quality drops — roughly past ~100k tokens. So we size tasks small and start fresh often. I'll show the token counter live. 'Clear vs compact': compacting keeps sediment; clearing returns to the model's sharpest state. The whole workflow that follows exists to keep us in the green zone.
06
How we stay in the smart zone (workflow + skills)
This is the map for the night. We stay in the smart zone by following a workflow, and each step is a "skill" — a packaged expert prompt you call with a slash. Tell the room: I already have these installed; you grab them by literally asking Claude "add the grill-with-docs / to-prd / to-issues skills from github.com/mattpocock/skills." Now we'll walk each one — what it does and why — then use them for real. align → destination → slice → build & test → verify.
07
/grill-with-docs — ALIGN
Matt Pocock's favorite skill — four sentences that change everything. The goal isn't a document, it's a SHARED DESIGN CONCEPT (Fred Brooks): you and the AI on the same wavelength. It asks relentlessly, recommends an answer, you steer. For beginners: a relentless, helpful teammate that asks the questions you forgot to. We'll run it live against our spec sheet. ("I needed to be on the same wavelength as my agent.")
08
/to-prd — DESTINATION
Matt: the "destination document." Because AI is great at summarizing, we trust it to summarize our alignment and we just skim. PRD = Product Requirements Document. Best parts for beginners: the user stories ("as a vendor, I want to scan a badge so that…") and the OUT-OF-SCOPE list (how you avoid scope creep and know when to stop). We'll generate it live straight from the grill.
09
/to-issues — SLICE
Matt's tracer-bullets idea (The Pragmatic Programmer): fire in the dark and you can't see where shots land — tracer rounds glow so you get feedback. AI loves to build HORIZONTALLY (all one layer), but then nothing works until the end. We force VERTICAL slices: thin, complete threads through the whole stack. Each becomes a small, grabbable task — and several can run in parallel. We'll slice our app live and pick the first one.
10
/tdd — BUILD & TEST
Matt: TDD is essential for getting the most out of agents — it adds good tests AND stops the AI from faking it (write code, then a test that rubber-stamps it). Your feedback loops are the CEILING on AI quality. We already used this on the one piece of real logic in our app — reading a name+email off a QR code (5 tests, all green). We can pop those open. Don't TDD everything (not the UI) — use it where there's logic worth trusting.
11
MCP — giving Claude tools
Skills = workflows; MCP = tools. Frame simply: "how you give Claude tools." Playwright MCP is the wow — the agent drives a real browser to QA the app (same as the obney ops workshop). Setup = one ask: "set up the Playwright MCP in this project" → Claude writes the config itself → restart Claude Code → /mcp confirms. (Prep tip: we pre-ran it so the browser's already downloaded, won't stall.) Use it live to QA the Scan slice.
